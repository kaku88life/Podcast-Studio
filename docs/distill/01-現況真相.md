## C. Pipeline 架構（文件裡怎麼設計的）

文件把產線拆成 **A–G 七環節**（對應概念上的 0–9 十階段）。核心金律有三條，值得記住：

1. **三軌對齊用 `cue_id` 不用秒數**：`voice.cues[]` 是唯一時間真值，視覺/字幕用 `cue_ref＋anchor＋offset` 綁「句」。語音逐句重生成後句長變，視覺自動跟隨，不用重標時間碼。（`workflow-integrated.md` D1）
2. **閘門由狀態機強制**：下游能被觸發 ⟺ 上游欄位已存在且已 approve。
3. **只卡 4 個人工覆核點**：B 腳本定稿 / C 配音試聽 / D 數據圖查證 / E 成片簽核；其餘自動鏈式推進。

| 環節 | 做什麼 | 主要技術（文件傾向） | 人工卡點 |
|---|---|---|---|
| A 選題＋捕捉 | 題目評分→兩人雙軌錄音→轉錄逐字稿 | **複用 line-inspiration-helper**（Whisper＋日文不動產詞庫） | 選題拍板 |
| B 腳本化 | 逐字稿→AI 整理成結構化對話（去卡詞、保口語、**只整理不創造**） | Claude（Opus 校準、Sonnet 量產，tool calling 強制 JSON） | **定稿閘門** |
| C 語音合成 | 克隆雙聲音＋腳本→自然對話音檔＋字級時間碼 | **ElevenLabs** v3 Text-to-Dialogue | **試聽閘門** |
| D 視覺生成 | 不出鏡：字幕/簡報/數據圖/B-roll，以字級時間碼為骨架 | **Remotion**＋Nano Banana Pro（圖內中文可靠） | **數據查證閘門** |
| E 組裝＋後製 | 音＋字幕＋視覺→分層渲染＋轉碼/響度 | Remotion＋FFmpeg（loudnorm −14 LUFS） | **成片簽核閘門** |
| F 轉化＋發佈 | 長片/Shorts/Podcast RSS/blog/社群＋AI 揭露 | Vizard＋Postiz＋自架 RSS | **發佈前強制勾 AI 揭露** |
| G 資產管理（貫穿） | Episode 為唯一真值、狀態機、job queue、版本/血緣 | FastAPI＋Celery＋PostgreSQL＋MinIO/R2 | 守 4 閘門 |

**與 line-inspiration-helper 複用**：環節 A 捕捉層直接接既有 `/api/captions` 轉錄＋SRT/VTT＋詞庫（雙軌分軌、日文不動產專有名詞已設計，成本約 $0.006/分鐘）。證據：`workflow-architecture.md` L222–231（main）。

## D. Episode 資料模型（已定義）

`Episode` 物件八層：`ideation` / `capture` / `script` / `voice` / `visuals` / `assembly` / `outputs` / `publish`，每層再展開欄位（例：`script.dialogue[]` 帶 `speaker/text/emotion/pause_after_ms/pronunciation_overrides/origin`；`voice.cues[]` 是時間真值）。DB 方針：狀態/標題/時間戳正規化（方便看板索引），script/voice/visuals 隨環節演進的區塊放 **JSONB**（迭代不改 schema）。

狀態機：`ideation → captured → scripted_pending_review →(人 approve) scripted → voiced_pending_review →(人 試聽) voiced → visualized_pending_review →(人 補圖) visualized → assembled_pending_review →(人 簽核) assembled → published`。證據：`workflow-integrated.md` L147–167（feat）、`workflow-architecture.md` L100–146（main）。
