# 蒸餾 02 — 目標 Spec 與願景

> 產出日期：2026-07-05｜這份把散在 repo／Obsidian／life-os 的願景收斂成一條「乾淨 pipeline」＋使用者故事，作為之後開發的北極星。

---

## 一、願景一句話

**兩人（Master＋學弟 Hiro）用「自然閒聊」當原料，AI 當放大器，把日本不動產知識做成 Podcast／YouTube；真人不必出鏡、不怕卡詞，但每個 AI 環節都保留人的覆核點。**

- 主題：日本（關西/大阪為主）不動產知識型內容，主軸「建商＋區域＋物件」逐集深化。
- 目標受眾：想買日本房產、對日本不動產有興趣但缺系統知識的台灣人。
- 商業目的：知識型內容建立專業品牌 → 帶來看房/買房客源。
- 產出形式：不出鏡（虛擬形象/簡報式視覺）＋克隆聲音＋字幕＋數據圖表。
- 節奏：先跑通第一集半自動流程，穩定後目標「一週一次或一月 3–4 次」。

證據：`Meetings/2026-05-29-…會議記錄`、`Sources/2026-06/…Session筆記-Podcast-Studio`、`Wiki/Projects/Podcast-YouTube/*`。

## 二、乾淨 Pipeline（捕捉 → 腳本 → 配音 → 影片）

把七環節收斂成使用者實際會走的四大棒，每棒標「輸入 / AI 做什麼 / 人在哪卡 / 輸出」：

### 棒 1 — 捕捉 Capture（複用既有，幾乎 0 開發）
- 輸入：兩人一段輕鬆的雙軌錄音。
- AI：接 line-inspiration-helper 的轉錄（Whisper＋日文不動產詞庫）→ 逐字稿＋SRT/VTT。
- 人卡：選題拍板、逐字稿詞庫快掃。
- 輸出：`capture.transcript` ＋ `segments[]`（帶 `source_start/end/speaker`）。

### 棒 2 — 腳本 Script（新，1 個 LLM 呼叫＋編輯介面）
- 輸入：逐字稿。
- AI：Claude 把口語逐字稿整理成結構化對話腳本——**清卡詞/贅詞、補邏輯、保留口語風，但嚴守「只整理不創造觀點」紅線**。情緒/停頓標記在這一棒標（有原音上下文）。
- 人卡：**定稿閘門**——逐行 approve 才放行下一棒。
- 輸出：`script.dialogue[]`（`approved: true`）。
- 模型策略：**首集用 Opus 校準品質，之後常態用 Sonnet 量產。**

### 棒 3 — 配音 Voice（新，串 ElevenLabs API）
- 輸入：定稿腳本。
- AI：ElevenLabs v3 Text-to-Dialogue，用兩人克隆聲音生成自然對話音檔＋字級時間碼。
- 人卡：**試聽閘門**——逐句試聽、調情緒/停頓。
- 輸出：`voice.lines[]`＋`voice.cues[]`（唯一時間真值）＋`master_audio_url`。
- **合規前置：Hiro 書面同意書必須先簽（見合規節）。**

### 棒 4 — 影片 Video（新，Remotion 模板）
- 輸入：配音音檔＋字幕＋數據圖表。
- AI：Remotion 簡報式模板，以字級時間碼為骨架自動排版字幕/簡報/圖表；FFmpeg 轉碼＋響度正規化（−14 LUFS）。配圖傾向 Nano Banana Pro（圖內中文可靠）。
- 人卡：**數據查證閘門**（價格/年份/出處人工核對）＋**成片簽核閘門**。
- 輸出：YouTube 長片（`outputs.youtube`），之後再擴 Shorts/Podcast/blog。

> **時間軸關鍵設計**：整條線用 `cue_id` 綁句、不用絕對秒數。任何一句配音重生成，句長變了視覺也自動跟隨，省掉「改一句就要重標整條時間軸」的痛。

## 三、使用者故事（User Stories）

以「Master／Hiro／未來的便宜模型代理」三種角色寫，方便驗收。

**選題與企劃**
- 作為 Master，我想在看板把選題卡從「選題池」拖到「企劃確認」，這樣兩人對進度一目了然。（✅ board.html 已可做）
- 作為 Hiro，我想在工具確認表對每個工具自己評分、寫試用筆記，這樣工具由我們試完決定，不是被 AI 挑掉。（✅ tools-checklist.html 已可做）

**捕捉**
- 作為 Master，我錄完一段閒聊後，想一鍵得到帶日文不動產專有名詞正確的逐字稿與字幕，不用重打。（🔜 接 line-inspiration-helper）

**腳本**
- 作為 Master，我想讓 AI 把逐字稿整理成順的對話腳本，但保留我們的口語風、且絕不幫我們捏造沒講過的觀點；我要能逐行改、逐行核可才往下走。（🔜 M2）

**配音**
- 作為 Hiro，我先簽一份同意書授權克隆我的聲音；之後腳本定稿就能生成我「沒卡詞版」的聲音，我要能逐句試聽、不滿意就重生成該句。（🔜 M3，前置＝同意書）

**影片**
- 作為 Master，我想把音檔＋字幕＋我補的數據圖丟進 Remotion 模板就自動排成一支影片；牽涉價格/數據的圖，我要能在發佈前逐項核對出處。（🔜 M4）

**發佈與合規**
- 作為 Master，發佈前系統要強制我勾「本片含 AI 合成聲音」揭露，沒勾不讓發。（🔜 F 環節，MVP 平台級揭露）

**產能（未來）**
- 作為 Master，一份成片 master 我想一鍵切出 Shorts、Podcast 音檔、blog、社群短文。（🔜 M5，MVP 之後）

## 四、MVP 邊界（先做什麼、先不做什麼）

**MVP＝跑通「一集」半自動端到端**（`workflow-architecture.md` MVP 定義）：
1. 輕鬆錄音 → 轉錄＋字幕（複用，0 開發）
2. AI 腳本化 → 人工定稿（新）
3. ElevenLabs 雙聲音合成（新）
4. Remotion 簡報式影片模板（新）
5. 人工覆核 → 匯出 YouTube 長片

**MVP 刻意先不做**：Shorts 自動切、多平台自動發佈、虛擬主播、自動配圖全自動、波形編輯器 Stage 2–4、產線全自動 Celery 編排（先手動串）。

**技術決策（文件已給傾向，可直接沿用）**：
- 產線編排：主幹自建 FastAPI＋Celery；n8n 只做 M6+ 無狀態雜務。
- 聲音：雙軌並行——先零樣本/Instant 跑通 MVP，平行錄 Professional 語料。
- Remotion 渲染：MVP 本機，量產再切 Lambda。
- 物件儲存：MinIO 開發、R2 上線。
- Episode DB：狀態/metadata 正規化＋環節區塊 JSONB（Supabase 現有 `episodes` 表可沿用起步）。
- 詞庫：獨立 `glossary_japan_re.txt`（日本不動產專有名詞）。

## 五、聲音克隆合規 spec（硬性，內建進流程）

1. **書面同意書**：正式開始克隆 Hiro 聲音前，簽一份書面同意（授權範圍＝本頻道內容、可撤回、素材用途）。建議存 Drive `00_ADMIN/法務帳務_Legal-Finance`。
2. **平台 AI 揭露**：每次發佈強制勾平台揭露（YouTube `containsSyntheticMedia`、IG「Made with AI」）；影片描述加一句「本集使用 AI 語音合成，基於兩位主持人真實討論製作」。
3. **內容底線**：只輸出「兩人真實討論過、已人工覆核」的觀點，不無中生有、不假造數據。
4. **系統面（建議加）**：Episode 資料模型加 `voice.consent`（who / date / scope / doc_url）與發佈前的「揭露已勾」硬性檢查。

> 這四點是紅線，不是選項。棒 3（配音）在同意書簽好之前不得啟動。
